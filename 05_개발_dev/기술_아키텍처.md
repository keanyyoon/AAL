# 기술 아키텍처 (Technical Architecture)

- 작성일: 2026-07-01 · 작성: AAL 개발팀 · 범위: Q4 파일럿 MVP 기준(과확장 금지) + 확장 훅
- 성격: 스택 후보·비교·권고. 최종 스택은 팀 역량·기존 자산(HUMMINGo 등)과 함께 확정 필요.

## 1. 원칙
- **파일럿에 맞는 최소 복잡도** + **확장을 막지 않는 데이터/구조 결정**만 선반영.
- 4대 데이터 로깅 무결성이 최우선(파일럿 목적).
- 반응형 웹(태블릿·PC primary, 모바일). 네이티브 앱 아님(학부모 알림은 카카오/웹푸시).

## 2. 시스템 구성 (논리)
```
[클라이언트: 반응형 웹 SPA]  학생 세션 · 성장맵 · 학부모 리포트
        │ (HTTPS/REST or tRPC, WS for 대화 스트리밍)
[BFF/API 게이트웨이]  인증 · 요청 라우팅 · 로깅 훅
        ├── [학습 세션 서비스]  지문/문항 제공, 응답 처리, 상태
        ├── [AI 오케스트레이션]  소크라틱 발문·비계·페르소나 톤 (LLM 호출·가드레일)
        ├── [로깅/이벤트 파이프라인]  세션·문항·페르소나·사고행동 이벤트 → 데이터스토어
        └── [리포트 서비스]  데일리 로그·4주 추이·AI 코멘트 생성
[데이터]  관계형(사용자·세션·문항결과) + 이벤트 로그 스토어 + (확장) 지식그래프
[외부]  LLM API, 카카오 알림, 청담 계정 연동
```

## 3. 스택 후보 (권고)
| 레이어 | 후보 | 권고/메모 |
|---|---|---|
| 프론트 | React + TypeScript (Next.js) | 반응형·i18n 성숙, 컴포넌트 재사용 |
| 스타일 | 디자인 토큰 기반(CSS 변수/Tailwind) | 브랜딩 토큰 공유, 다크모드·다국어 대비 |
| 백엔드 | Node(NestJS) 또는 Python(FastAPI) | AI 오케스트레이션 친화(택1, 팀 역량 기준) |
| DB | PostgreSQL | 관계형 + JSONB(유연 로그), 확장 시 그래프 분리 |
| 이벤트 로그 | append-only 이벤트 테이블 → (확장)분석 DW | 파일럿은 단순, 확장 시 스트리밍 |
| AI | LLM API + 자체 프롬프트/가드레일 계층 | 페르소나 톤·소크라틱 로직은 앱 계층에서 제어 |
| 인증 | 청담 SSO 연동 + 간편 로그인 | 아동 개인정보 최소 |
| 알림 | 카카오 알림톡 + 웹푸시 | 성과 시에만 |

## 4. AI 오케스트레이션(요약, 상세는 소크라틱_AI_설계.md)
- 공통 발문/비계 로직(과목 비종속) + 페르소나 톤 파라미터 주입 + **정답 유출 가드레일**.
- 대화는 스트리밍(WS/SSE), 모든 턴에 페르소나 ID·사고행동 태그 로깅.

## 5. 확장 훅 (아키텍처)
- **i18n**: 문자열·콘텐츠·튜터 발화 리소스 분리(하드코딩 금지).
- **사고요소 층 분리**: 논리·지식② 언어독립 층을 별도 스키마로(국어·수학 재사용).
- **외부준거 매핑 테이블**: 내부레벨 ↔ 외부기준 데이터화.
- **멀티리전/결제 골격**: 인터페이스만 예약(파일럿 미구현).
- **성장 로그 학생 단위 영속**: 콘텐츠 교체와 무관하게 누적.

## 6. 비기능 요구
- 성능: 대화 응답 체감 지연 최소(스트리밍). 접근성 AA. 보안·개인정보(아동) 준수. 로그 무결성(완료버튼=트랜잭션 확정).

> **결정 요청:** 백엔드 언어(Node vs Python)와 LLM 제공자 방향. 기존 자산(HUMMINGo·지식그래프) 연동 범위 확인 필요.
