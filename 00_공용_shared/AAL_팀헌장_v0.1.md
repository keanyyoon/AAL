## 3. 협업 모델 — 서로 논의하고 최적안을 찾는 방식

Agent는 상시 상주하지 않는다. **오케스트레이터(나)** 가 각 팀 Agent를 실행하고, 산출물을 공용 폴더에 모으고, 팀 간 논의를 중재한다. 유저는 **지시·컨펌·최종 완결**을 결정한다.

### 3.1 워크플로우 (Round 구조)
```
[유저 지시]
   ↓
Round 1  리서치 → (기획) → (UX·브랜딩·개발 병렬) → 검증 교차리뷰
   ↓  각 팀 초안 산출물
[크로스 리뷰]  검증팀 정합성 리포트 + 팀 간 상호 코멘트
   ↓
[유저 컨펌/피드백]  ← 여기서 방향 확정
   ↓
Round 2  피드백 반영·심화 → 최적안 수렴
   ↓
[유저 최종 완결]
```

### 3.2 의존성(누가 누구를 기다리나)
- 리서치 → 기획 (기획은 리서치 위에)
- 기획(PRD) → UX·개발·브랜딩 (스코프 단일 진실 소스)
- UX ↔ 브랜딩 (톤·비주얼 상호 정합), UX → 개발
- 브랜딩(페르소나) → 개발(페르소나 로깅)
- 전 팀 → 검증 (마지막 교차 리뷰)
- **병렬 가능**: UX·브랜딩·개발은 기획 PRD 확정 후 동시 진행

### 3.3 팀 간 논의 규칙
- 이견은 `00_공용_shared/논의로그_decisions.md`에 **논점 / 각 팀 입장 / 근거 / 결정(또는 유저 에스컬레이션)** 형식으로 기록.
- 검증팀이 불일치를 발견하면 관련 팀에 반영 요청 → 미해결 시 유저 에스컬레이션.
- 최적안 판단 기준(우선순위): ① 파일럿 4대 데이터 확보 가능성 ② 교육 철학(소크라틱·문해력) 부합 ③ Q4 개발 현실성 ④ 확장 훼손 없음 ⑤ 학생 몰입/재미.

### 3.4 나(오케스트레이터)와 유저의 역할
- **오케스트레이터**: 팀 실행·조율, 산출물 취합, 정합성 관리, 유저에 요약·에스컬레이션 제시.
- **유저(Youngmin)**: 각 Round 시작 지시, 산출물 컨펌/수정 지시, 최종 완결.

## 4. 공용 규칙 (Ground Rules)

1. **폴더 구조** (워크스페이스 내)
   - `00_공용_shared` (헌장·논의로그·용어집) / `01_리서치` / `02_기획` / `03_UX디자인` / `04_브랜딩` / `05_개발` / `06_검증` / `99_참고문서`
2. **언어**: 한국어 + 핵심 용어 영어 병기.
